# Agent Targets and Managed Control Plane Boundary

Bachkator should add first-class Agent Targets inside the OSS CLI contract, while private or enterprise governance should live in a managed control plane that consumes Bach evidence. The OSS boundary is the Bachfile syntax, CLI commands, JSON outputs, logs, quality reports, agent reports, and policy results. The private control plane owns organization state, prompt and policy registries, provider credentials, workspace provisioning, approvals, audit exports, usage analytics, and cross-project findings history.

Agent orchestration belongs in Bach when it needs normal target behavior: typed addresses, dependencies, dry-runs, logs, cache evidence, quality gates, completion contracts, and State Store records. An `agent` should therefore be a Target kind handled through the shared Target model rather than macro-generated shell targets or a separate agent-run universe. Providers supply execution mechanics, but the Agent Target owns Bach semantics such as workspace clone, branch and commit requirements, improvement attempts, policy attachment, and agent report ingestion.

Policies are named acceptance contracts above individual quality reports. A policy may require targets such as `group.gate`, reviewer Agent Targets, and quality gates over normalized findings. An improvement loop is distinct from retry: retry reruns the same target attempt, while improvement feeds findings and policy evidence back to the implementation agent until the policy passes or the configured attempt limit is exhausted.

The first implementation slice stayed local and OSS: support an implementation-mode Agent Target, a `provider` block, clone workspace defaults under `.bach/agents/<name>`, required git commit behavior, and a structured agent report that can be parsed into quality findings. Later slices added review-mode agents, reusable policies, improvement attempts, merge-mode agents, and JSON evidence exports while preserving the OSS/control-plane boundary.

Atelier or another private control plane should integrate through Bach's CLI and JSON contracts instead of importing internal Go packages or reading the private State Store schema. That split keeps Bach useful as a local and CI execution engine while letting the enterprise product build the durable moat: organization policy, Agent Packs, prompt approvals, managed OpenCode workspaces, model governance, cross-repo findings, cost and token telemetry, approval workflows, and evidence over time.
